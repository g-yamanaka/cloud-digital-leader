# Google Cloud Digital Leader 用語まとめ

## ストレージ（Cloud Storageとクラス）

### Nearline
- 低頻度アクセス向けの低コストオブジェクトストレージクラス
- 月に数回程度アクセスするバックアップやログ保管に向きます。頻繁アクセス向けより保存単価は下がる一方、取り出し料金や最低保存期間などの条件を意識します。

### Coldline
- さらにアクセス頻度が低いデータ向けの超低コストストレージクラス。
- 災害対策用バックアップや長期保管データに向きます。保存は安いですが、取り出し料金や最低保存期間などの制約がより強めです。

### Archive
- ほぼ取り出さない前提の最安クラス長期保管ストレージクラス。
- 監査・法令対応の証跡保管や年単位で眠らせるデータに向きます。取り出しは可能ですが、コストや所要時間の面で「いざという時だけ」を想定します。

### Cloud Storage
- あらゆるファイルをバケットに保存できるオブジェクトストレージ。
- 画像・動画・ログ・バックアップなどを高い耐久性で保存し、スケールも自動です。ライフサイクル管理やIAMで運用・コスト最適化しやすいです。

### Requester Pays
- Cloud Storageで、データの取得/リクエストにかかる費用を「アクセスした側」に負担させる課金設定。
- 公開データ配布などで配布元のコストを抑えるのに使えます。想定外の請求を避けるため、利用者側の設定・案内が重要です。

## データベース（マネージド/データストア）

### Firestore
- アプリ向けのマネージドNoSQLドキュメントデータベース。
- 柔軟なスキーマで保存でき、モバイル/Webアプリと相性が良いです。リアルタイム更新やセキュリティルールによりクライアント直結の設計がしやすいです。

### Cloud Bigtable
- 低レイテンシで大規模スループットを扱える、ワイドカラム型のNoSQLデータベース。
- 時系列データや大規模ログ、IoTなどに向きます。キー設計が性能に直結するためアクセスパターン前提で設計します。

### Cloud Spanner
- グローバル分散と強整合性を両立するマネージドのリレーショナルデータベース。
- 水平スケールしながらトランザクション整合性を保てます。高可用性やグローバル展開が必要な基幹系で候補になります。

### Cloud SQL
- MySQL / PostgreSQL / SQL Server をマネージドで提供するリレーショナルデータベース。
- 既存RDBの延長で使いやすく、運用（パッチ、バックアップ等）を軽くできます。一方でスケール要件が大きい場合は別選択肢も検討します。

## サーバーレス（実行基盤）

### Cloud Run
- コンテナをサーバーレスで実行できるマネージドサービス。
- HTTPリクエストに応じて自動スケールし、使った分だけ課金されます。APIやWebhook、軽量なバッチ処理などに向きます。

### Cloud Functions
- 1つの関数をイベント駆動で実行するサーバーレス実行環境。
- Pub/Sub、Cloud Storage、HTTPなどのイベントをトリガーにコードを実行できます。小さな自動化や「つなぎ込み」の処理に向きます。

---

## コンテナ/Kubernetes/ハイブリッド

### Google Kubernetes Engine
- Kubernetesクラスタをマネージドで運用できるサービス。
- コンテナのデプロイ、スケール、ローリング更新、自己修復をKubernetesで管理できます。マイクロサービス基盤や可搬性を重視する構成で選ばれやすいです。

### Anthos
- ハイブリッド/マルチクラウド環境を統合管理するプラットフォーム。
- オンプレや他クラウドを含む複数環境で、ポリシーや運用を統一できます。段階的な移行や環境差分の吸収に強いです。

### マイクロサービスアーキテクチャ
- 1つの大きなアプリを、独立デプロイ可能な小さなサービス群に分割する設計スタイル。
- チーム分割やスケールを柔軟にしやすいです。その代わり分散システムの運用（通信、監視、障害分離）が難しくなります。

---

## ハイブリッド/VMware/ベアメタル

### Google Cloud VMware Engine
- Google Cloud上でVMware環境（vSphereベース）をマネージドに提供するサービス。
- 既存のVMを大きく作り替えずにクラウドへ移行しやすいです。VMware運用の延長でクラウド利用を進められます。

### VMware vSphere
- 仮想化基盤（ESXi + vCenterなど）を中心としたVMwareの代表的なプラットフォーム。
- 仮想マシンの作成・運用・管理を統合的に行います。オンプレの既存資産で採用されていることが多いです。

### ベアメタルソリューション（Bare Metal Solution）
- Googleが提供する専有の物理サーバー環境（ベアメタル）を利用できるサービス。
- 仮想化では要件を満たしにくいワークロード（特定ライセンス/低レイテンシ等）に向きます。クラウド近接でオンプレ相当の実行環境を確保できます。

### ハイパーバイザー
- 物理マシン上で仮想マシンを動かすための仮想化レイヤ（例：Type1/Type2）。
- リソース（CPU/メモリ/IO）を分離して複数VMを同居させます。vSphereの中核要素（ESXiなど）としても登場します。

## API管理

### Apigee
- APIの公開・保護・分析を行うAPI管理プラットフォーム。
- 認証/認可、レート制限、APIキー、監査や分析を一元的に提供します。社内外にAPIを安全に提供し、利用状況の可視化も行えます。

## データ分析/パイプライン/BI

### BigQuery
- 大規模データをSQLで分析できるサーバーレスDWH。
- インフラ管理なしで高速に集計・分析できます。ログ分析、ELT、BI連携の中核になりやすいです。

### Dataflow
- バッチ/ストリーミング処理を統合できるデータ処理サービス。
- データ変換・集計・ETL/ELTをスケールさせて実行できます。Pub/Sub→BigQueryなどのストリーミングパイプラインでよく使われます。

### Looker
- 指標定義を統一して分析を提供するBIプラットフォーム。
- セマンティック層で指標や集計ロジックを管理し、解釈ブレを減らせます。ダッシュボードや探索分析をガバナンス付きで提供できます。

### データウェアハウス
- 分析用途に整形したデータを集約して活用する仕組み。
- 意思決定のための集計・分析を高速に行うことを目的とします。構造化データ中心で、整合性とクエリ性能を重視します。

### データレイク
- 生データを形式を問わず大量に貯める保管・分析基盤。
- 構造化/非構造化の両方を取り込み、後から用途を広げられます。メタデータ管理や統制が弱いと検索性・品質が落ちやすいです。

## メッセージング/イベント駆動

### Pub/Sub
- メッセージを非同期に配信するマネージドメッセージングサービス。
- PublisherとSubscriberを疎結合にしてイベント駆動を実現できます。システム連携やストリーミング処理の入口としてよく使われます。

## AI/ML

### AI/ML
- 機械学習モデルの作成を支援するマネージド機能群。
- データと目的に合わせて学習・評価・デプロイまでの流れを簡略化できます。モデル品質はデータ品質に強く依存する点が重要です。

### Vertex AI
- 生成AI/機械学習の開発と運用を統合するプラットフォーム。
- 学習、推論、評価、MLOpsを一貫して扱えます。生成AIでは安全性や運用監視、プロンプト管理も含めて設計します。

### AutoML
- 機械学習モデルの作成を自動化・簡略化するための機能群（一般に特徴量設計〜学習の一部を支援）。
- まずは素早くモデルを作って精度感を掴みたい時に有効です。最終品質はデータ前処理や目的設計に左右されます。

### Dialogflow
- チャットボット/音声ボットの対話フローを構築できる会話型AIプラットフォーム。
- 意図（インテント）と会話状態を設計し、問い合わせ自動化などに使えます。外部システム連携で実用性が大きく上がります。

### TensorFlow
- 機械学習モデルを構築・学習・推論するための代表的なOSSフレームワーク。
- モデル実装や学習パイプラインの基盤として使われます。運用では再現性（データ/コード/パラメータ管理）が重要です。

### TPU
- 機械学習（特に深層学習）の計算を高速化するための専用アクセラレータ。
- 大規模学習や推論のコスト/時間を改善できます。利用可否はモデルやフレームワーク対応など要件確認が必要です。

## DevOps/成果物管理

### Artifact Registry
- コンテナイメージやパッケージ（各言語の成果物）を保管・配布するためのマネージドレジストリ。
- CI/CDで作った成果物の保管場所として使われます。権限管理や脆弱性対応など運用面の統制にもつながります。

## 移行（マイグレーション）

### Migrate for Compute Engine
- 既存環境（オンプレ/他クラウドなど）のVMをCompute Engineへ移行するための支援サービス。
- 段階移行（テスト→切替）を取りやすく、移行作業の標準化に寄与します。ネットワーク/権限/依存関係の事前整理が成否を分けます。

### Migrate for Anthos
- 既存アプリ/VMなどをコンテナ化してAnthos（Kubernetes基盤）へ寄せる移行を支援するサービス群。
- “まず動かす”から“コンテナ運用へ最適化”までの橋渡しになります。移行後の運用設計（CI/CD、監視、ポリシー）が重要です。

## 可観測性（監視/トレーシング）

### アウト・オブ・ザ・ボックスの可観測性
- 導入直後から標準機能だけで監視・観測ができる状態。
- メトリクス・ログ・トレースが最初から揃っていて状況把握をすぐ始められます。一方で観測の粒度や制約を理解し、必要なら追加計装します。

### Cloud Trace
- 分散トレーシングでリクエストの遅延要因を可視化するサービス。
- マイクロサービス等の多段呼び出しで、どこがボトルネックか追いやすくなります。ログ/メトリクスと合わせて原因切り分けを高速化します。

## ネットワーク

### Virtual Private Cloud（VPC）
- Google Cloud上に作る論理的なプライベートネットワーク（サブネット、ルーティング、FWなどを含む）。
- リソース間通信の境界を定義し、セキュリティ/分離を実現します。設計次第でオンプレ接続や複数環境の分離にも効きます。

---

## アイデンティティ/認証/ディレクトリ連携

### LDAP
- ディレクトリサービスに問い合わせるためのプロトコル（ユーザー/グループ情報の参照など）。
- 企業内の認証基盤（Active Directory等）と関係が深いです。クラウド連携ではディレクトリ同期やSSO設計の前提知識になります。

### Identity Platform（REST API）
- アプリ向けの認証基盤（IDaaS）を提供し、REST APIで認証機能を組み込みやすくするサービス/インターフェース。
- ユーザー登録、ログイン、トークン発行などをアプリ側で扱いやすくします。認証フローや権限設計は別途ポリシーとして設計します。

### Google Cloud Directory Sync
- オンプレのディレクトリ（例：Microsoft ADなど）とGoogle側のディレクトリを同期するためのツール。
- 既存のユーザー/グループ管理を活かしてアカウント運用を移行・統合しやすいです。ID管理の一貫性を保つのに有効です。

## 信頼性/運用

### 信頼性エンジニアリング（SRE）
- 信頼性を数値目標で管理し、運用を工学的に改善する考え方/職能。
- SLOとエラーバジェットを軸に、開発速度と安定性をバランスさせます。障害対応だけでなく自動化と再発防止で運用負債を減らします。

### SLA
- 提供者が保証するサービス品質の取り決め（契約）。
- 稼働率やサポート条件などが明記されます。未達時のクレジット（返金相当）がセットで示されることが多いです。

### SLO
- サービス品質の目標値（運用上のゴール）。
- 可用性やレイテンシ、エラー率などを数値で定義します。SLAより内向きの目標として、運用判断や改善の軸になります。

## ガバナンス/コンプライアンス

### コンプライアンスレポートマネージャー（Compliance Reports Manager）
- 監査・規制対応に必要なGoogle Cloudのコンプライアンス関連ドキュメントを探しやすくする機能/ポータル。
- 各種認証（例：ISO、SOCなど）に関するレポートへのアクセスを整理できます。監査準備や証跡提示の効率化に役立ちます。
